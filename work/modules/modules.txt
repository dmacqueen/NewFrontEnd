Thoughts on modules in MsML.

Guiding principles

-- Simplicity and "obviousness"
   of individual "features" and of interactions between features.
   As far as is possible, features should be independent (*orthogonal*)
   The term "obvious" here means "nearly canonical, once one has properly understood
   the concept", which understanding may not be achieved immediately on first encounter.

-- If we can declare (i.e. name) something, we should be able to (lambda-) abstract over
   such a name. This is according to

    Landin's Correspondence Principle
      [Stated by Landin in The Next 700 PLs, CACM 1966; named by Tennant, in Acta Informatica, 1977]

    [Tennant, 1977] This discipline [Correspondence and Abstraction principles] helps
    in both discovering and correcting unnecessary restrictions, "missing" facilities,
    irregularities in the syntax, semantics, or terminology, unexpected interactions or
    complications, and so on.

where "This discipline" refers to the Correnspondence and Abstraction principles in that paper. 

* Concepts (and terminology)

-- Modules as encapsulated (reified), named environments

The most basic idea of a module is that a module is a named piece of environment (both static
and dynamic).  We will concentrate primarily on the static side here.  We call the named
piece of environment a *structure*.

-- Type of a module -- *signatures*

Given that we are starting with a typed "value language", there is a natural notion of the
"type" of an environment, which gives rise to the idea of a *signature* of an environment,
which simply means that natural *type* of the environment.

-- environments are "mixed" or "heterogeneous"

Environments contain bindings for a mixture of static (e.g. type) and value components.

The module system is roughly a typed lambda calculus over (heterogenious) environments,
with "signatures" (environment types) as types.


-- Hierarchy

We extend the basic notion of a (mixed) environment by allowing bindings of structures (named
environments) to be included in environments, and hence allow a structure to contain a named
structure component (a *substructure*).

"Environments can contain bindings of named environments, i.e. modules."

-- Terminology
   Historical, in the design of the Standard ML module system. Has a *mathematical* flavor.
   Simple module (named environment) called a "structure" (suggested by Universal Algebra).
   Module function or *abstraction* called a "functor" (borrowed from Category Theory).
   Module type is called a "signature" (from modern algebra).

   Alternative, more conventional names from PL culture might be "module" (for structure),
   "module function" (for functor), and "module type" (for signature).

-- Generative functors
   - as opposed to "applicative functors" in Caml (CAML?).


* Design (syntax and operational semantics)
  -- An environment is the denotation of a *declaration*

  -- Statically "first-class" structures vs "second-class" structures
     (or SML '90 sharing semantics vs SML '97 sharing semantics).

     Is a structure just a container of tycons? In which case (1) a structure that doesn
     not contain any type declarations is *degenerate* and has no *static* content.  (This
     is the SML '97 semantics).

     Or does (2) a structure itself have a static identity (as in the SML
     '90 semantics), regardless of whether it contains (transitively) any type (tycon)
     components?

     In case (2), all structures have a static identity and are hance subject to sharing
     specifications. This has an interesting consequence when the structure contains
     *state*, since sharing specifications can enforce identity of two stateful structures.
     Also, stucture declarations and functor applications generate new structure
     identities (the "nonapplicative" functor semantics).
     

* Implementation


* Beyond modules - the meta level of modularity

CM demonstrated the need to manage collections/groups/libraries of modules.  CM's
"metalanguage" for managing these things seems to based on or heavily influenced by
older tools like Unix *make*.  We need to consider whether a layer of notations for
this sort of management should be part of MsML or external, in a modernized CM-like
tool.


* Simplified module syntax in MsML

- no abbreviations, shortcuts, or derived forms

For instance, no implicit functor parameter structures
Instead of writing

    functor F ( type t val x : t) = struct ... t ... x ... end 

where an internal ghost parameter name like <paramStr> or <F.param> is
used, one has to write (more explicitly)

    functor F (X : sig type t val x : t end) = struct ... X.t ... X.x ... end

And so on with other syntactic shortcuts or abbreviations.


* Development of module system as 2-nd level typed lambda calculus

- There should be a direct, explicit syntax for expressing functor signatures.

   E.g. (X: sig1) => sig2

  This is a 2nd level *dependent* product (or dependent function space) notation.
