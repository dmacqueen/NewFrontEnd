The problem of designing and implementing the next generation of CM (Compilation Manager) is
peripheral to the main task of the NewFrontEnd/MsML project, but at some point it will
be relevant.  And it is major issue relating to how MsML can build on the existing SML/NJ.

This document is a very preliminary draft that may develop into a useful treatment of
these issues. Many of the points or topics below are interrelated, and the order of the
points does not necessarily reflect any dependency between topics or concepts. Later
refinement of this document may introduce a more logical ordering or unification of
topics.

Note: In the beginning, the REPL was assumed as the context in which SML code would be
compiled and run. (the main designers only had experience of languages (e.g. Lisp, Pop 2)
where this was the case.  CM and binary libraries came along later and were developed in
an *ad hoc* way in parallel with the build process for the SML/NJ compiler.

20. Difference in computational model behind "conventional" batch-compiled languages like
    C, Pascal, etc. and a functional language like SML.
    - In "conventional" imperative languages, code and data are rather distinct, while in
      SML, code and data "coexist" in a rather intimate combination (such as function
      closures).
    - The actual pieces of code (as byte arrays, "blocks of bytes" in the heap), can be
      treated specially (like strings) in a multi-arena G.C.  code blocks do not contain
      any pointers, for instance.
    - data, whether local or global, normally is represented by pointer structures in the
      garbage-collected heap.
