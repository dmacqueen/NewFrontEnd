Some general observations about types in MsML (and (Standard)ML)

* types vs type constructors (types vs tycons)
  - Types are expressed by "type expressions" built from type variables using tycons
  - We don't distinguish between type type "int" and the type constructor called "int".
    i.e. nullary type constructors like "int" are treated as types

  - [Implementation]. In the implementation (as in SML/NJ), they may be different.
    int : TYPE
    int : () => TYPE, int () : TYPE

* structural typing (type equivalence)
  - primitive types
  - products (tuples, records)
  - sums (delegated to datatypes)
  - nominal typing (what does it mean in the context of ML? comparison with generativity)
    -- nominal typing differs from typeing with generative type declaraions, in that
       the generative identity of a tycon remains even if we have lost that ability to
       name the tycon (e.g. because of a shadowing declaration)
    -- this may imply that structural type identity cannot be captured by purely syntactic
       means
    -- See the notion of "type explicitness" from the SML Commentary.

* generativity of type declarations
  - A new, unique static identity for a type constructor is *generated* at its declaration. 
    -- this identity is "robust" and does not depend on the ability to *name* the tycon
  - datatypes
  - opaque signature matching (the new "abstract" types)
  - [related to generativity of structure declarations and *signature instantiation* in the
     module system]
  - a tycon may have multiple names or no name

* type equivalence (when do two types "match" or "agree" in type checking)
  - structural modulo generativity (and inductive tycons like list)
  - treatment of *inductive* tycons like list (t == t' ==> list t == list t')
  - MsML does not allow the definition of *coinductive* tycons (??)

* datatypes
  - tagged sum, recursion, parameterized
  - datatypes as modules [CMU view]

* structural equality
  - certain primitive types, e.g. int, string
  - equality preserved by certain type constructors
    -- products
    -- lists
  - datatypes that *preserve equality*
    i.e. produce an equality type when applied to an equality type
    -- e.g. bool, list
    -- can be recursive (inductive)
  - types with infinite values (function types, streams) are not equality types
    -- i.e. function types and *coinductive* types?
  - abstract types are not equality types
    -- even if their *interface* (whatever that might be) contains an equality function for
       the type
    -- even though a function might be called "eq" or "equal" or "same", it is not necessily
       an equality (or "the" equality) for the type.  Same for "compare".
  - type variables (are abstract, and) are not equality types
  - orderings ("lt", "lessThan", "compare") are not generic and always have to explicitly
    provided.

* relation betweeen type level and value level
  - related to "phase distinction"
  - no way to "derive" values from types (no type classes!)

* types exist independent of how they are named
  - a shadowed type declaration does not mean the type no longer exists
  - nameless types (types that (no longer) have a name

* [implementation] types and space usage

  Some implemenation of the SML type system or typed intermediate
  languages (TILT, FLINT in particular) have encountered problems with the space used by
  type representations. Space usage problems appear to be associated with duplication
  of type structure when reducing applications of type functions, including the generic
  instantiation of polytypes. To increase sharing of type structure and avoid duplication,
  TILT used let-binding or naming of shareable type structure, while FlINT use
  hash-consing of type structure.

  There is a question about whether types used in intermediate representations such as
  FlINT (where the flint IR is based on F_omega) use more space for type representations
  than the Front End type system.  There does not seem to be an obvious reason why IR
  types should consume more space than FE types.

  In SML/NJ, type reduction (i.e., application of tyfuns) used a simple device to avoid
  copying type subterms that were not affected by physical substitution.  The file
  work/types/type-funs.txt describe another, roughtly equivalend approach, involving a sort
  of compilation of type functions.
  
* kinds

  Type constructors are characterized by a kind, which reduces to the number of
  type arguments they take.  So kinds of type constructors are all of the form

    TYPE # ... # TYPE => TYPE

  and if there are n (>= 0) type arguments, the type constructor is of *arity* n.
  A null-tuple of types is denoted by "()".

  Type constructors of arity 0 are typically implicitly applied (to 0 arguments), so

    int : () => TYPE      -- int as a type constructor
    int == int () : TYPE  -- int as a type

* type declarations

  There are two sorts of type declarations.  These declaration introduce/create nameed
  type *constructors*, where 0-ary type constructors are usually treated as type constants
  (e.g. int, bool)

  (1) type functions.

  E.g.

     type point = int * int
     type 'a pair = 'a * 'a

  These are first-order, of kind TYPE* => TYPE with arity given by the number of type
  arguments (where all arguments are of kind TYPE, as explained above).  Type functions
  are first-order and are not recursive, so reduction of an application of a type function
  (by beta-reduction) is trivial.  In principle, type functions whose definition is known
  can always be eliminated by reduction.


  (2) (algebraic) data types (inherited from the Hope language)

  E.g.
  
    datatype bool = True | False

    datatype 'a list = Nil | Cons (of) 'a * 'a list

  Datatype declarations introduce an atomic, irreducible type constructors of some specified
  aritys (arity of bool = 0, arity of list = 1).  Datatypes declarations are *generative*,
  in that a new static identity is generated whenever such a declaration is elaborated.

  There is no *dynamic* semantics associated with type constructor declarations.
  I.e. such declarations do not participate in the dynamic semantics of the language.

  By convention, the names of data constructors of datatypes are alphanumeric with an
  initial capital letter [with some "historical" exceptions, like true, false, nil, ::.

* Abstract type constructors

* equality of types

  In principle, equality of types is term equality after reducing away all type function
  constructors (with datatype and abstract type constructors being irreduciable).

  In practice, equality can be made more efficient by making type functions be
  *semi-transparent*, i.e assigning an (weak) identity to them

* names of type constructors

  Type constructors are not nominal. That is, we do not determine whether two type
  constructors are equal by their names. This is because

    - a type constructor may *loose* its name, e.g., through shadowing.

    - a type constructor may be *renamed*. [?]

  When a type constructor *looses* its name, it still has its internal static identity,
  and may still be a component of the type, say, of a variable or constructor.  This
  differs from the SML Definition, with its notion of "type explicitness", [see
  Commentary, X.x, page ?].

  We don't require that a declaration within the scope of another type declaration declare
  a different name (thus we allow shadowing).  This is because of the traditional way the
  environment in a REPL behaves, where we can re-enter or re-load declarations multiple
  times.
